Подвиг 8 (развитие подвига 7). Необходимо объявить класс-декоратор с именем Handler, который можно было бы применять к функциям следующим образом:

@Handler(methods=('GET', 'POST')) # по умолчанию methods = ('GET',)
def contact(request):
    return "Сергей Балакирев"

Здесь аргумент methods декоратора Handler содержит список разрешенных запросов для обработки. Сама декорированная функция вызывается по аналогии с предыдущим подвигом:

res = contact({"method": "POST", "url": "contact.html"})

В результате функция contact должна возвращать строку в формате:

"<метод>: <данные из функции>"

В нашем примере - это будет:

"POST: Сергей Балакирев"

Если ключ method в словаре request отсутствует, то по умолчанию подразумевается GET-запрос. Если ключ method принимает значение отсутствующее в списке methods декоратора Handler, например, "PUT", то декорированная функция contact должна возвращать значение None.

Для имитации GET и POST-запросов в классе Handler необходимо объявить два вспомогательных метода с сигнатурами:

def get(self, func, request, *args, **kwargs) - для имитации обработки GET-запроса
def post(self, func, request, *args, **kwargs) - для имитации обработки POST-запроса

В зависимости от типа запроса должен вызываться соответствующий метод (его выбор в классе можно реализовать методом __getattribute__()). На выходе эти методы должны формировать строки в заданном формате.

P.S. В программе достаточно объявить только класс. Ничего на экран выводить не нужно.

Схема и код получились примерно такими:

class Decorator:
    def __init__(self, a, b):
        self.a = a
        self.b = b

    def __call__(self, func):
        def wrapper(x, y=1):
            return func(x) + y * (self.a + self.b)
        return wrapper


# @Decorator(4, 8)
def my_func(x):
    return x

dec_obj = Decorator(4, 8)   # Объявляем объект класса-декоратора
my_func = dec_obj(my_func)  # Атрибут занятый функцией my_func превращается в функцию wrapper
res = my_func(10)  # Аргумент 10 передаётся в функцию wrapper функции-декоратора __call__ класса Decorator
print(res)

По схеме можно сделать интересные выводы:

  • Роль функции-декоратора выполняет метод __call__;
  • Функция my_func превратилась в функцию-обёртку wrapper;
  • Класс-декоратор Decorator, получается, не такой уж и декоратор! Всю роль взяла на себя __call__, да и через @ пишется не класс, а объект класса - @Decorator(4, 8)

 Параметр* ни во что не превращается, он принимает ссылку на функцию my_func и передаётся методу __call__, который теперь берёт на себя всю работу по декорированию.

То есть, без аргументов, мы использовали __init__как decorator, а __call__как wrapper. C аргументами же, нам необходима дополнительная обёртка, которой мы обернём декоратор, чтобы передать ему аргументы. Ею теперь становиться __init__, которому мы передаём аргументы, принимаемые декоратором  @decorator(*args).

То есть, __init__ выступает в роли обёртки для декоратора, единственная задача которой - принимать аргументы декоратора (а не декорируемой функции).

Для наглядности, вот реализация декоратора с параметрами (вне контекста классов-декораторов и функторов):

def df_decorator(dec_param):
    def decorator(func):
        def wrapper(s):
            s = func(s) + dec_param
            return s
        return wrapper
    return decorator


@df_decorator(dec_param=5)
def int_sum(dec_param):
    s = sum(map(int, dec_param.split()))
    return s


print(int_sum('1 2 43 56'))

Вот мои 5 копеек в обсуждении этой задачи. 
Если переписать образец декорирования (из описания задачи)

@Handler(methods=('GET', 'POST')) # по умолчанию methods = ('GET',)
def contact(request):
       return "Сергей Балакирев"
    
без синтаксического сахара, то вызов декорированной функции будет выглядеть так

contact = Handler(methods)(contact)
contact(request)

Первая строка - декорирование, вторая строка вызов декорированной функции.
Handler(methods) - это экземпляр класса Handler. В __init__ класса Handler мы передаем список methods для инициализации этого экземпляра. И больше ничем его не инициализируем. Привет self.
К этому экземпляру можно обратиться как к функции. Тогда произойдет вызов метода __call__. Вот он этот вызов

Handler(methods)(contact)

Мы передаем ему (вызову) аргумент contact. Аргумент contact - это функция, которую надо декорировать. 
Но для этого во внутрь __call__ надо передать еще аргумент request. Как? Методы __init__ и __call__ уже использованы. Вот для этой передачи и создается оберточная функция wrapper, которой и передается аргумент request. Так что структура класса Handler должна быть такой.

class Handler:
      def __init__(self,methods):
           self.methods = methods

      def __call__(self,func):
           def wrapper(self,request):
            ---- Код ---
            return func(request)
            ---- Код ---
       return wrapper

Впрочем Балакирев об этом и говорит в Небольшой справке.
Еще одно замечание. Внутри wrapper функция contact должна вызываться, поскольку наружу мы возвращаем функцию wrapper. Функция wrapper тогда станет декорацией функции contact. При вызове wrapper произойдет вызов contact.

Ну и теперь итоги.
Handler(methods)  - это объект класса Handler. 
Handler(methods)(contact) - вызов этого объекта, что приводит к вызову метода __call__, которому передается функция contact. Вызов __call__ вернет функцию wrapper. Ну и вызов функции wrapper
        
Handler(methods)(contact)(request)

вернет требуемый результат.

@Алекс_Блинов, он нужен чтобы вызывать метод get или post, передавая название метода. Например:

self.__getattribute__('get')(func, request)

Название метода берется из словаря request по соответствующему ключу. Именно для этого и нужны два метода get и post, которые просят реализовать, чтобы они были доступны для такого вызова. По крайней мере я это условие понял так.
