Подвиг 3. Объявите класс с именем Clock и определите в нем следующие переменные и методы:

- приватная локальная переменная time для хранения текущего времени, целое число (своя для каждого объекта класса Clock с начальным значением 0);
- публичный метод set_time(tm) для установки текущего времени (присваивает значение tm приватному локальному свойству time, если метод check_time(tm) возвратил True);
- публичный метод get_time() для получения текущего времени из приватной локальной переменной time;
- приватный метод класса check_time(tm) для проверки корректности времени в переменной tm (возвращает True, если значение корректно и False - в противном случае).

Проверка корректности выполняется по критерию: tm должна быть целым числом, больше или равна нулю и меньше 100 000.

Объекты класса Clock предполагается использовать командой:

clock = Clock(время)

Создайте объект clock класса Clock и установите время, равным 4530.

@Elena_Vedeneva, это просто некое соглашение. Создавая класс, мы создаём некий новый тип данных, с каким-то предсказуемым поведением для пользователя. В каких-то случаях, если пользователь не соблюдает правила игры, выгодно поднять шум, запустить исключение и т.д. В каких-то случаях, даже если правила игры не соблюдены, бывает приемлемо промолчать. Например, пусть у нас есть две таблички данных в Pandas, с разным числом строк, и нам нужно посчитать сумму элементов построчно:

import pandas as pd
df1 = pd.DataFrame({'col1': [1, 2]})
df2 = pd.DataFrame({'col1': [3, 4, 5]})
df1 + df2

Ответ будет состоять из 3х строк, третья строка будет содержать NaN (потому что мы как бы сложили 5 неизвестно с чем). Никаких ошибок, никаких исключений. Означает ли это, что создатели Pandas не знают правила хорошего кода на питоне? Да нет, конечно, просто бывает удобно выбирать правила игры по ситуации. Если у нас математическая задача с матрицами, в которых пропущены элементы - весь смысл задачи скорее всего пропадает, логично запускать исключение и падать. Если же нам попался набор данных с дырками, а в жизни они почти всегда с дырками, это правило, а не исключение. Запускать по этому поводу исключение неуместно как-то.
